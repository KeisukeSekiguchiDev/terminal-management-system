# 事業・システム構想ドキュメント
## TechCore Solutions TMS開発プロジェクト

**文書バージョン**: 1.0
**作成日**: 2025年11月23日
**ステータス**: ドラフト

---

## 1. プロジェクト背景

### 1.1 現状の課題
- **コスト課題**: PayConnect等の外部TMSサービス利用によるランニングコストが高い
- **技術力向上**: 自社開発により内部の技術力・ノウハウを蓄積する必要性
- **柔軟性確保**: 外部サービスでは実現できない独自機能の実装ニーズ

### 1.2 開発判断の経緯
- 当初はPayConnect TMS 2.0の利用を検討
- コスト分析の結果、AWS/Azure上での自社開発の方が長期的にコスト効率が良いと判断
- 自社の技術力向上と差別化機能の実装を目指す

---

## 2. プロジェクト目的（Why）

### 2.1 主要目的
1. **コスト削減**: 外部サービス利用費の削減（目標: 50%以上のコスト削減）
2. **技術力向上**: 社内でのTMS開発・運用ノウハウの蓄積
3. **差別化機能**: 競合他社にない付加価値機能の実装
4. **拡張性確保**: TC-200以外の端末にも対応可能な柔軟な設計

### 2.2 ビジネス目標
- 端末管理コストの最適化
- 新規端末導入時の迅速な対応
- 顧客満足度の向上（障害対応時間の短縮）

## 3. スコープ定義

### 3.1 In Scope（対象範囲）
| カテゴリ | 内容 |
|---------|------|
| **対象端末** | ・TechCore Solutions TC-200（初期対応）<br>・将来的な他メーカー端末への拡張性を確保 |
| **主要機能** | ・端末登録・認証管理<br>・ファームウェアOTA更新<br>・アプリケーション配信<br>・パラメータ設定変更<br>・リモート鍵注入（RKI）<br>・端末監視・診断<br>・レポート・分析<br>・差別化機能（後述） |
| **ユーザー** | ・システム管理者<br>・運用オペレーター<br>・加盟店管理者<br>・サポート担当者 |
| **システム基盤** | ・クラウド（AWS or Azure）<br>・Web管理画面<br>・REST API |

### 3.2 Out of Scope（対象外）
- 決済処理そのものの機能
- 端末ハードウェアの製造・修理管理
- 在庫管理システムとの連携（Phase 1では対象外）

## 4. 想定ユーザーとペルソナ

### 4.1 ユーザーセグメント

| ユーザータイプ | 役割 | 人数規模 | 利用頻度 |
|---------------|------|----------|----------|
| **システム管理者** | TMS全体の管理、設定 | 2-3名 | 毎日 |
| **運用オペレーター** | 日常的な端末管理、監視 | 5-10名 | 毎日 |
| **加盟店管理者** | 自店舗の端末管理 | 100-1000店舗 | 週1-2回 |
| **サポート担当者** | 障害対応、問い合わせ対応 | 3-5名 | 毎日 |
| **開発者** | API利用、システム連携 | 5-10名 | 随時 |

### 4.2 主要ペルソナ

#### ペルソナ1: 運用オペレーター（田中さん）
- **背景**: 24時間体制で端末監視を行う運用チーム
- **ニーズ**:
  - 異常をすぐに検知したい
  - 一括で複数端末を操作したい
  - 障害対応手順を簡素化したい
- **ペインポイント**:
  - 現在は手動確認が多く時間がかかる
  - 複数システムを見る必要がある

#### ペルソナ2: 加盟店管理者（鈴木店長）
- **背景**: 複数店舗を管理する店舗オーナー
- **ニーズ**:
  - 端末の状態を簡単に確認したい
  - トラブル時にすぐサポートを受けたい
- **ペインポイント**:
  - 技術的な知識が限定的
  - 忙しい中で対応が必要

## 6. 差別化戦略（付加価値機能）

### 6.1 競合分析に基づく差別化ポイント

| 差別化機能 | 説明 | 優位性 |
|-----------|------|--------|
| **AI予測保守** | 機械学習による故障予測 | 競合の多くは事後対応のみ |

### 6.2 MVP vs 将来機能

- 基本的な端末管理機能

- 基本的な監視・アラート

#### Phase 2: 差別化機能（6-12ヶ月）
- AI予測保守
- 高度な分析機能
- マルチテナント
- API公開

#### Phase 3: 先進機能（12ヶ月以降）
- 音声アシスタント
- ブロックチェーン連携
- IoT統合

## 7. 技術スタック（推奨）

### 7.1 アーキテクチャ選定理由

| レイヤー | 技術選択 | 選定理由 |
|---------|---------|----------|
| **バックエンド** | Java/Spring Boot | ・エンタープライズ実績豊富<br>・セキュリティ機能充実<br>・大規模対応可能 |
| **API層** | REST + WebSocket | ・RESTは標準的<br>・WebSocketでリアルタイム通信 |
| **フロントエンド** | React + TypeScript | ・コンポーネント再利用<br>・型安全性<br>・豊富なライブラリ |
| **データベース** | PostgreSQL + Redis | ・RDBMSとキャッシュの併用<br>・高性能 |
| **インフラ** | AWS | ・実績豊富<br>・マネージドサービス活用 |
| **監視** | CloudWatch + Datadog | ・詳細な監視<br>・アラート機能 |

### 7.2 初心者向け学習パス
1. **基礎知識**（2週間）
   - Web技術の基礎（HTTP、REST API）
   - データベース基礎（SQL）
   - クラウド基礎（AWS）

2. **実装技術**（4週間）
   - Spring Boot チュートリアル
   - React 基礎
   - Docker 入門

3. **実践開発**（継続的）
   - ペアプログラミング
   - コードレビュー
   - 段階的な機能実装

## 8. リスクと対策

| リスク項目 | 影響度 | 発生確率 | 対策 |
|-----------|--------|----------|------|
| **技術力不足** | 高 | 中 | ・外部専門家の活用<br>・段階的な開発<br>・充実した研修 |
| **セキュリティ脆弱性** | 高 | 低 | ・セキュリティ監査<br>・PCI DSS準拠<br>・定期的な脆弱性診断 |
| **スケール課題** | 中 | 中 | ・クラウドネイティブ設計<br>・負荷テスト実施 |
| **コスト超過** | 中 | 低 | ・段階的リリース<br>・コスト監視 |
